GC HealthLink - Actual System Architecture & AI Logic
=====================================================

Based on the actual code implemented in the workspace (specifically the frontend/backend directories and ai-service-python/main.py), the system utilizes a modern microservices architecture and focuses on epidemiological forecasting and resource management.

1. Actual System Architecture (Microservices)
---------------------------------------------
The system is built on a decoupled, three-tier microservice architecture:

* Frontend (User Interface): Built with Next.js 15 (React) using Tailwind CSS for styling. It handles the dashboards for the medical and dental sides.
* Core Backend (Data API): Built with Node.js and uses Prisma ORM to connect to the primary database. This handles user authentication, patient records, and standard CRUD operations.
* AI Microservice: A standalone Python application built with FastAPI. It exposes predictive endpoints and communicates with the main Node.js backend. It uses pandas for data manipulation and scikit-learn for machine learning.


2. Actual AI Logic & Predictive Analytics
-----------------------------------------
The AI service handles two specific predictive domains. It currently focuses on Epidemiological Forecasting and Resource Management:

A. Outbreak Forecasting Model (/predict/outbreak)
This logic predicts future spikes in specific illnesses (e.g., flu, dengue) among the student population.
* Algorithm Used: Linear Regression via scikit-learn.
* How it Works: 
  1. Data Aggregation: The API receives historical health records (dates, illness categories, and case counts). It uses pandas to floor the dates to the start of the month and aggregates total cases per month for each illness.
  2. Time-Series Modeling: It extracts the month index as the "time feature" and the case counts as the target variable.
  3. Prediction: It fits a stable linear trend model to the historical data. It then predicts the exact number of cases for N months into the future. If an illness is trending upwards, the system will forecast higher numbers for upcoming months, alerting the clinic to prepare for an outbreak.

B. Resource Depletion & Inventory Prediction (/predict/resources)
This logic prevents the clinic from running out of critical supplies (e.g., Paracetamol, Amalgam capsules) using deterministic algorithmic prediction rather than ML.
* Algorithm Used: Rate-of-Consumption Mathematical Modeling.
* How it Works:
  1. Burn Rate Calculation: It takes the current_stock and the daily_usage_rate of an item and calculates exactly how many days are left until total depletion.
  2. Predictive Timeline: It calculates a concrete predicted_depletion_date (e.g., "Will run out on May 12th").
  3. Automated Risk Categorization: It classifies inventory into statuses:
     - Critical: <= 7 days of stock remaining.
     - Warning: Stock will run out within the user-defined horizon (e.g., 30 days).
     - Stable: Healthy stock levels.
  4. Smart Restock Recommendation: It automatically calculates a recommended_restock_qty targeting a 60-day buffer to ensure the clinic buys exactly what it needs without overstocking.
